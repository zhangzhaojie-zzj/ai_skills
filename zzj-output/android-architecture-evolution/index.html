03 / 原始架构

原始架构的困境

❌ 架构特点

  • AsyncTask + Handler
    两层结构：Data Layer + View Layer
  • Activity/Fragment 臃肿
    业务逻辑全部堆积在 UI 层
  • 回调地狱
    嵌套回调难以维护
  • 无法单元测试
    逻辑与 UI 深度耦合

⚠️ 核心问题

  • 代码结构丑陋
    难以阅读和理解
  • 维护成本高
    添加新特性容易出错
  • 职责不清晰
    数据与展示混杂
  • 测试困难
    无法独立测试业务逻辑
04 / MVC 架构

MVC：来自 Java Web 的尝试

Model
数据模型
→
Controller
控制器
→
View
视图层

✓ 优点

  • 耦合性降低
  • 可扩展性改善
  • 模块职责相对明确

! 局限性

  • Activity 仍包含 UI 代码
  • View 与 Controller 绑定
  • 仅部分优化效果
05 / MVP 架构

MVP：Google 的第一次尝试

Model
数据层
↔ 接口
Presenter
逻辑层
↔ 接口
View
UI 层

核心优势

Model 与 View 完全解耦 通过接口交互，松耦合 便于单元测试

缺点

代码量增加 需要编写大量接口
06 / MVVM 架构

MVVM：数据绑定时代

Model
数据模型
→
ViewModel
视图模型
⇄
View
视图层

通过 DataBinding 实现数据与 UI 的自动同步

✓ 优势

  • UI 层与业务逻辑隔离
  • 降低程序耦合度
  • 自动数据同步

! 挑战

  • 会产生很多类型
  • XML 和代码两处维护
  • 学习曲线较陡
07 / 响应式编程

RxJava 与 Agera

🔄 RxJava + RxAndroid

通过异步流的方式处理数据，提供丰富的操作符（map、filter、merge 等）

避免嵌套回调 简化代码 无生命周期管理

🆕 Agera（Google 官方）

类似 RxJava 的函数式响应编程，专为 Android 设计，支持生命周期感知

Observable
事件源
→
active/inactive
生命周期感知
→

⏳ 刚推出，待观察
08 / 总结

选型建议

✅

新项目推荐

选择 MVP 或 MVVM，结构清晰，便于维护，适合团队协作

✅

已有项目

可逐步引入 RxJava，优化异步处理，避免回调地狱

❌

避免使用

原始架构（已过时），AsyncTask 已废弃，代码难以维护

💡

选型原则

根据团队熟悉度选择 · 考虑项目复杂度 · 保持代码规范
